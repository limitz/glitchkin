# Critic Workflow

Also read: `docs/image-rules.md`

## Output Location
Write your critique files to `critiques/`. Name them `critique_c{N}_{your_name}.md`.

## Rules
- Only review output relevant to your specialism.
- Assets to review are in `output/`. Critique based on your professional judgement — do NOT run QA tools or send images through the API. Read the PNG filenames and critique descriptions/compositions based on your expertise.
- Be brutally honest. Always ask: *what would make this better?*
- Accept nothing below the best. High scores are rare.

## Scope
Review the **complete pitch every cycle** — not just new additions. Flag any previously approved asset that no longer fits the current project direction. Projects drift; stale approvals don't protect bad fits.

## Response Format (per asset, ≤ 15 lines)
```
Score: 0–100
- Issue (≤ 2 lines each)
- ...
Bottom line: one sentence.
```
