**Idea:** Add a `--known-issues PATH` flag to `LTG_TOOL_ci_suite.py`. This flag accepts a JSON file listing expected WARN/FAIL results by check name and item. When a known issue matches, the suite reports it as "EXPECTED WARN" rather than a plain WARN, reducing noise for pre-existing false positives (e.g. G007 on non-Glitch files, W004 draw order advisories). The exit code would only count non-expected failures. The known-issues JSON would live alongside `arc_diff_config.json` and `glitch_spec_suppressions.json` — same pattern, easily updated without touching Python source. This is the CI counterpart to glitch_spec_lint's suppressions.json, which has already proven effective at reducing persistent false-positive noise.
**Benefits:** Reduces cognitive overhead for everyone reading CI suite output. Currently the C39 suite output has 26 draw-order advisory WARNs and 12 glitch-spec WARNs, all pre-existing. Anyone scanning for real regressions has to read through known noise. A `--known-issues` flag would let the suite produce a clean "0 unexpected WARN, 0 FAIL" result for stable baselines, making true regressions immediately obvious.
