**Idea:** Add a per-asset false-positive exception registry to LTG_TOOL_render_qa_v001.py. Currently, the same WARN flags appear every cycle on well-audited assets (SUNLIT_AMBER on Luma/Miri sheets, UV_PURPLE on SF03, warm_cool on all single-key style frames), and each cycle a human has to re-verify they are documented false positives rather than real regressions. A simple JSON or Python dict in the tool mapping asset filenames (or asset types) to known-exempt check+color combinations would allow the tool to annotate those results as "FP-DOCUMENTED" instead of "WARN", and exclude them from the WARN count. The QA report would still list them (for transparency) but mark them green. New WARNs — ones NOT in the registry — would immediately stand out as genuine issues requiring investigation. This would cut C31 QA analysis time by ~60% and reduce the risk of missing a real regression buried among known false positives.
**Benefits:** Sam (faster audit each cycle), Alex (cleaner grade counts to report), future critics (fewer confusing WARN entries in QA reports). The registry doubles as living documentation of known tool limitations, which is valuable for onboarding.
