**Idea:** Extend `LTG_TOOL_render_qa._check_warm_cool()` to support an `expected_temp` parameter with values "warm", "cool", "contested", or None (default = current top-vs-bottom median split). For "warm" mode: PASS if >= 40% of saturated pixels fall in the warm family (0-65° and 300-360°). For "cool": PASS if >= 40% fall in 120-240°. For "contested": PASS if both warm >= 5% AND cool >= 5%. For None: current behaviour preserved. The `qa_report()` function could accept an asset metadata dict that specifies `expected_temp` per asset, or the check could be suppressed via `asset_type="single_temp_frame"`. This came directly from the C35 warm/cool root cause analysis — all 4 pitch SFs fail the current check despite having intentional, correct temperature design because the metric tests vertical split rather than temperature intent.
**Benefits:** Eliminates 4 systematic WARN flags on all style frames (SF01=warm, SF02=cool-dominant, SF03=cold, SF04=warm) that have been documented as false positives since C31 but never fixed at the source. Would reduce noise in the pre-critique QA pipeline and allow Priya Nair's warm/cool critique dimension to be properly gated: the tool would only flag frames where the dominant temperature is wrong, not frames where the temperature is uniform-by-design.
