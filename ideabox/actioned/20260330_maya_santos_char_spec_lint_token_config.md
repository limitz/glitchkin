**Idea:** Make `LTG_TOOL_char_spec_lint.py` M004 (and similar token-based checks) data-driven via a config file rather than hardcoded token lists. Currently M004 validates Miri hair accessory by searching for the strings "chopstick", "bun", "CHOPSTICK", "BUN", "chop_stick" in the generator source. When the chopstick accessory is renamed to wooden hairpins (pending Alex Chen confirmation, C44), the token list must be updated in sync with the tool rename — if one is updated without the other the lint either false-positives or false-negatives. A small JSON or TOML config mapping each lint check ID to its current token set would decouple the two, making it safe to update spec terminology without simultaneously touching the lint tool, and making the lint rules readable to non-developers.
**Benefits:** Reduces cross-file coordination risk for any future canonical design terminology change (not just the Miri hairpin swap). Kai Nakamura (owner of char_spec_lint) could implement the config loader in a single cycle. Would benefit the whole team any time a naming or design vocabulary decision ripples through multiple files.
