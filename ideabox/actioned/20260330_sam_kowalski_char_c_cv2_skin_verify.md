**Idea:** Now that OpenCV (cv2) is authorized (Alex Chen broadcast C40), upgrade the color verify tool to use LAB ΔE for skin tone verification instead of Euclidean RGB distance. The current `verify_canonical_colors()` tool uses radius=40 in RGB space, which causes systematic false positives on Cosmo and Luma skin swatches flagging as SUNLIT_AMBER hits. CIELAB ΔE is perceptually uniform — a ΔE threshold of ~8-10 would correctly distinguish skin tones from amber lamp colors that are visually distinct but numerically close in RGB. The CHAR-C addition this cycle surfaces the same problem: any QA run on Cosmo character sheets will trigger identical false positives until the tool is upgraded. Proposed API change: `verify_canonical_colors(path, mode="lab")` where `mode="lab"` uses `cv2.cvtColor` → LAB space → ΔE distance, and `mode="rgb"` preserves existing behavior.
**Benefits:** Eliminates the entire SUNLIT_AMBER false-positive class (FP-003 through FP-005 in qa_false_positives.md) for all character sheets containing Luma, Cosmo, and Miri skin tones. Would allow Kai Nakamura to remove 5 documented false-positive exceptions from the CI registry, reducing per-cycle QA investigation time for Sam and Morgan Walsh. Directly applicable to CHAR-C entries added this cycle.
