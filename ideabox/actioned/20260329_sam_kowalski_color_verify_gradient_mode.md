**Idea:** Add a "hue histogram" output mode to LTG_TOOL_color_verify.py. The current tool samples pixels within an Euclidean RGB radius of each canonical color, then takes the median hue. This produces false positives in two recurring cases: (1) gradient/anti-aliased scenes where many edge pixels surround canonical values at shifted hues — pulling the median off-target despite the canonical value being present exactly; (2) the "warm orange family" false positive, where GL-07 Corrupt Amber, RW-03 Sunlit Amber, and Luma's hoodie-UV values all share RGB neighborhood space. The fix: instead of returning just the median hue and a pass/fail, the tool should return a histogram of sampled hues showing how many pixels are at each hue angle. Then the verifier can see: "327 pixels at 271.9° (PASS), 120 pixels at 262° (gradient edges)" rather than a single median that hides the distribution. This would eliminate all three documented false-positive cases without changing the tool's architecture. Kai Nakamura could implement this as v002 of the tool in an hour — it is a pure reporting change, no core logic change needed. This would make every future color audit take 5 minutes instead of 30 minutes of false-positive investigation.
**Benefits:** Kai Nakamura (tool author) — minimal implementation work; Sam Kowalski — no more audit time wasted on false-positive investigation; critics (Priya Nair especially) — more credible QC documentation; the whole team — faster and more reliable automated color gating.
