**Idea:** Add a documented helper function `flatten_rgba_to_rgb(img)` to `LTG_TOOL_render_lib` that composites an RGBA image over a solid white/black background and returns a clean RGB image. In-generator warm/cool passes applied to RGBA buffers fail silently because accumulated semi-transparent alpha from prior composites interferes with Porter-Duff blending results — the effective pixel color diverges from what any external tool reading the saved PNG would see. A canonical flatten utility would prevent this class of bug entirely. Discovered during C41 tech den v006 work: the cool bottom pass (alpha=130) was numerically correct but produced wrong hue values because the RGBA base had non-255 alpha pixels; numpy Porter-Duff on the final RGB solved it. This pattern will recur in any multi-pass RGBA generator applying a final tonal correction pass.
**Benefits:** Saves future environment artists debugging time (this cost ~3 iterations to diagnose). Makes the generator-side tonal correction pattern reliable and documented. Relevant to Rin Yamamoto (style frames have similar multi-pass RGBA workflows), Morgan Walsh (CI baseline), and any future Hana environment work.
