## P1 — char_spec_lint Miri Filename Pattern Fix

**Root cause confirmed:** `_CHAR_REGISTRY` used `LTG_TOOL_grandma_miri_expression_sheet_v*.py` (with `_v*` suffix glob). The canonical file on disk is `LTG_TOOL_grandma_miri_expression_sheet.py` (no version suffix). The glob never matched it, causing M001–M005 to always report WARN/no-generator-found for Miri.

**Fix applied in v1.3.0:**
- Added no-version-suffix canonical patterns for all 4 characters (not just Miri — luma, cosmo, and byte had the same latent issue):
  - `LTG_TOOL_grandma_miri_expression_sheet.py` — the P1 fix
  - `LTG_TOOL_luma_expression_sheet.py` + `LTG_TOOL_luma_turnaround.py`
  - `LTG_TOOL_cosmo_expression_sheet.py` + `LTG_TOOL_cosmo_turnaround.py`
  - `LTG_TOOL_byte_expression_sheet.py`
- Legacy `_v*.py` versioned patterns retained for backward compatibility
- `_resolve_generators()` updated with deduplication (`seen` set) to prevent double-counting if a no-suffix and versioned glob both match the same file
- Comments in `_CHAR_REGISTRY` note the no-version-suffix convention for these canonical files

**Verification (as requested):** All other character generators (luma, cosmo, byte) confirmed — luma and cosmo have `_turnaround.py` + `_expression_sheet.py` (no suffix, same convention). Byte has `_expression_sheet.py`. Miri is NOT the only exception — the no-suffix convention applies to all 4 canonical generators. Only storyboard/scene tools use versioned suffixes.
