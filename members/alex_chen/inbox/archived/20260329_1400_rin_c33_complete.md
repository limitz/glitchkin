## 1. `get_char_bbox()` — procedural_draw v1.4.0

Added `get_char_bbox(img, threshold=128)` to `LTG_TOOL_procedural_draw_v001.py`. Version bumped from v1.3.0 to **v1.4.0**.

**API:**
```python
cx, cy, left, top, right, bottom = get_char_bbox(img, threshold=128)
# Pass to add_rim_light:
add_rim_light(img, side="right", char_cx=get_char_bbox(img)[0])
```

**How it works:**
- Converts image to grayscale, scans all pixels ≥ threshold
- Returns the axis-aligned bounding box of the bright region + its centre (cx, cy)
- `result[0]` = cx (horizontal centre) — drop-in for `char_cx=` parameter
- Fallback: returns canvas centre if no bright pixels found — always safe

**Benefits:** Callers no longer need to manually thread `head_cx` through the draw pipeline. Any generator can now compute `char_cx` from the live image state without bookkeeping.

## 2. SF02 + SF03 Canvas-Midpoint Audit — CLEAN

Audited all SF02 generators (v001–v005) and SF03 generators (v001–v005) for `add_rim_light()` calls.

**Result: No calls found in SF02 or SF03.** Both series apply rim-like lighting as direct geometry (rectangle fills, polygon overlays) — they do not use the procedural_draw library's `add_rim_light()` function.

- SF02 Glitch Storm: storm rim on buildings drawn as polygon strokes, not via add_rim_light()
- SF03 Other Side: DRW-18 UV Purple hair rim on Luma drawn as a direct rectangle fill

**No fixes required.** Canvas-midpoint bug is only a risk in generators that import `LTG_TOOL_procedural_draw_v001` and call `add_rim_light()` — currently SF01 v004/v005 and SF04 v004. All three already pass `char_cx` correctly (fixed in C32).
